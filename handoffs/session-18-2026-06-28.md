# cc-orchestrator ‚Äî 2026-06-28 ‚Äî session 18

Shipped **v1.9.0 ‚Äî fork into a live interactive session.** Fixes a design gap, not a crash:
forking from the dashboard ran `claude -p --resume <id> --fork-session` (headless one-shot via
`sendPrompt`, `/api/send`). The job ran and exited, writing a new forked session that landed
**idle** and never "running". A SemVer MINOR (new capability).

## What shipped

### A ‚Äî Fork ‚á¢ terminal (local: a real running interactive fork)
The terminal-attach helpers are now fork-aware. `attachInTerminal`/`attachCommand`
(`lib/actions.mjs`) take a `fork` flag that appends `--fork-session`, and `/api/attach`
(`server.mjs`) threads `fork: Boolean(fork)` through to both. A local-only **"Fork ‚á¢ terminal"**
button on each session card calls `attach(id, null, true)`, opening
`claude --resume <id> --fork-session` in a real Terminal ‚Äî a live interactive fork, not a
single headless turn. Gated behind `!IS_REMOTE` (a phone can't open a Mac Terminal).

### B ‚Äî Actionable fork jobs (universal, works on the phone)
After a headless fork job completes, `renderJobs()` makes the `forked ‚Üí <id>` line actionable:
**Open** (`openChat`), **Continue** (`openSend(id, false)`), and **Terminal** (`attachFork`,
local-only). New `attachFork(forkId, parentId)` helper opens the fork in Terminal, falling
back to the parent's cwd when the fresh fork isn't in the board index yet. Continue and
attachFork both pass `fork=false` ‚Äî they resume the existing fork id and must never re-fork.
New `.jobacts` flex row for the button group.
