## 2. Why this is easy: the data already exists

Nothing new needs parsing. The whole cost/usage pipeline is built and tested; the Usage tab is
mostly a **presentation layer** over functions that already return exactly these numbers.

### Token accumulation — `lib/cost.mjs`
- `sessionUsageByModel(path)` → `{ [model]: { input, output, cacheRead, cacheWrite5m, cacheWrite1h } }`
  for one transcript. This **is** the per-type breakdown the tab needs. Cached on `(size, mtime)`
  with byte-offset resume for live files.
- `usageByDateModel(path)` → `{ [YYYY-MM-DD]: { [model]: {…same token shape…} } }` — per-day × per-model.
- `rollupFromDaily(dailyMaps, { window, pricing })` → `{ window, buckets: [{ date, usd, byModel }], totalUsd }`;
  `window` ∈ `day` | `week` | `month`.
- `mergeUsageByModel(dst, src)` → merges two byModel maps additively (used to fold sub-agents in).
- `rollupToCsv(rollup)` → wide CSV export.

### Sub-agent spend — `lib/subagents.mjs` (this is the "separately" half of the ask)
- `subagentUsageByModel(projectsDir, projectDir, sessionId)` → byModel map for **only** the
  sub-agent transcripts of that session.
- `subagentDateModel(...)` → per-day × per-model for sub-agents.
- `subagentFilesFor(...)` → the sub-agent transcript list, `subagentCountFor(...)` → count.
- **Key invariant (already documented in `server.mjs` `attachCost`):** a parent transcript does
  **not** contain its sub-agents' assistant turns — the parent only carries a model-less Agent
  rollup on `user` lines, which the assistant-only accumulator never counts. So
  `own + sub` is **purely additive, no double-count**. Main-vs-sub is therefore a clean split:
  - main-session tokens = `sessionUsageByModel(<session>.jsonl)`
  - sub-agent tokens   = `subagentUsageByModel(...)`
  - total              = merge of the two
- Known gap (documented at the top of `lib/cost.mjs`): a Task-tool sub-agent whose turns live in a
  separate transcript records spend as a `toolUseResult.usage` rollup carrying **no model id**, so
  it isn't attributed. Measured impact ~0.1% of lifetime spend, ≤1.3% on the most sub-agent-heavy
  session. Don't chase it; just don't claim 100% precision.

### Pricing — `lib/pricing.mjs`
- `DEFAULT_PRICING` — USD per 1M tokens, per model. Override/extend via `~/.claude/contexts/pricing.json`.
- `rateFor(model, pricing)` — matches bare id, else **longest known prefix** (handles dated ids like
  `claude-haiku-4-5-20251001` and strips a `[1m]` window marker).
- `costSummary(byModel, pricing)` → `{ totalUsd, byModel: [{ model, usd, tokens, priced }], pricedKnown }`
  — sorted by usd desc. `priced:false` ⇒ that model contributes **$0** and `pricedKnown:false`
  (this is what renders the `*` on the header total).
- Cache multipliers live here and are the single source of truth:
  `CACHE_READ_MULT = 0.1`, `CACHE_WRITE_5M_MULT = 1.25`, `CACHE_WRITE_1H_MULT = 2.0`, all relative
  to the model's base **input** price.

### Context window / "over limit" inputs — `lib/scan.mjs`
- `WINDOW_BY_MODEL` + `contextWindowFor(model, desktopModel, usedTokens)`; each scanned session
  already carries `usedTokens` and its resolved context `window`. Per-session context-fill % is
  derivable from these without new parsing.

**Why the ledger exists (context that matters for a Usage tab):** cost is derived **live** from
transcript files with no persistence, so a deleted transcript silently drops its spend from the
"lifetime" total. Verified root cause of a real ~$34k → ~$23k drop the user noticed:
- `cleanupPeriodDays` is **unset ⇒ Claude Code's 30-day default deletes old transcripts.**
- `firstStartTime: 2025-07-28` but **oldest surviving transcript is 2026-05-18** ⇒ ~10 months of
  transcripts already auto-deleted. That spend is **unrecoverable locally** (it only exists in
  Anthropic Console billing now).
- The ledger stops future loss; it can only bank sessions it observes **while running**.

**Implication for the Usage tab: any "lifetime"/"all-time" number sourced only from on-disk
transcripts is wrong and will keep shrinking.** Either source it from the ledger, or label the
window honestly (e.g. "last 30 days of retained transcripts"). Do not print "lifetime" over
live-scan-only data.

## 6. Open question — resolve before building "over limit"

The user's phrase *"where are usage going form over limit model wise type usage"* is ambiguous.
Claude Desktop's `/usage` shows **subscription rate-limit consumption** (e.g. weekly limit %
per model). **cc-orchestrator has no access to that** — it reads local transcripts only and prices
them at public API rates as an **API-equivalent estimate**, not actual subscription billing.
So "over limit" cannot mean the Desktop rate-limit gauge unless a new data source is added.
Plausible local readings: per-session **context-window** fill (`usedTokens` / `window` from
`lib/scan.mjs` — this data exists), or a user-configured **budget** threshold
(`alerts.budget.thresholdUsd` in `lib/config.mjs` / `lib/alerts.mjs` — also exists).
**Ask the user which they mean; don't guess and build the wrong gauge.**
