## What this session did

Fixed the **silent build-staleness gap** discovered on 2026-06-24. The always-on process
(LaunchAgent `com.cc-orchestrator` :7433 / systemd `cc-orchestrator.service`) serves
`public/index.html` **fresh from disk** on every request, but its HTTP route table is **frozen at
process start**. So after a pull/merge that adds endpoints, the running process kept serving the new
UI (with new buttons) while `404`-ing the new `/api/*` routes those buttons call — until someone
manually restarted it. This bit the "Cost over time" button vs a ~4-day-old process (predated
`/api/cost/rollup`, `/api/history`, `/api/diag`); the frontend crash was already fixed in `0ecc92f`,
but the underlying drift remained and would recur on the next update.

Landed on branch **`claude/sleepy-ride-1ulg3e`**, commit **`39eed0b`** (pushed; **no PR opened** — the
task didn't ask for one). Implemented **all three** proposed options (they share one source of truth
and are all light):

### 1. Build/version stamp + UI mismatch banner (the core)
- **`lib/version.mjs`** (new) — the whole mechanism is server-side: capture the commit the process
  **booted** from (frozen at module load) and compare it to the commit on **disk now** (read fresh,
  3 s cache). They diverge exactly when code was updated under a live process.
  - `git rev-parse HEAD` via `execFileSync` with **cwd pinned to the repo root** (the service may be
    started from `/` or `~`), 2 s timeout, stderr ignored. **Unknown SHAs** (non-git / npm-tarball
    install, or no `git`) ⇒ `stale: false`, never a false alarm.
  - Exports: `versionInfo()` → `{ app, boot, head, stale, startedAt, uptimeSec, restart }`;
    `computeStale(boot, head)`; `restartCommandFor(platform)` / `restartCommand()` (pure, so every
    platform branch is unit-testable). **No version is baked into the HTML → still no build step.**
- **`GET /api/version`** in `server.mjs` (next to `/api/diag`).
- **`public/index.html`** — a **dismissible amber `#staleBanner`** (distinct from the red error
  `#banner`) shown when `stale`, carrying both SHAs + the **platform-appropriate restart command**.
  Key subtlety: a **`404` on `/api/version` is itself treated as stale** — the fresh disk HTML knows
  the route exists, so a process that 404s it must predate it. This makes the guard **self-detecting
  across its own introduction** (the very first deploy, where the running process has no
  `/api/version`, still raises the banner). Dismissal is keyed to the disk SHA in `sessionStorage`,
  so a *later, different* drift re-raises it. Wired into boot, the 20 s refresh interval, and
  `visibilitychange`.

### 2. `cc-doctor` staleness check
- `lib/onboarding.mjs` `runChecks` now probes an already-running server on `port` (`/api/version`,
  loopback ⇒ no auth) and **WARN**s when its in-memory build is older than `HEAD` (or 404s the
  route), with the restart command as the fix. Unreachable ⇒ no row (the port check covers "is it
  running"). No new network I/O in tests — `onboarding.test.mjs` never calls `runChecks`.

### 3. Docs + triage
- `README.md` — a Troubleshooting bullet for the exact symptom (`404 {"error":"not found"}` after
  updating) with per-OS restart commands, and a note that `cc-doctor` now WARNs on a stale build.
- `lib/diag.mjs` — `sanitizedEnv()` now includes `build` (boot short SHA) + `stale`; the bug-report
  issue body gets a `Build:` line flagging `⚠️ STALE`. A short git SHA is a public commit id, not a
  secret; no new PII.
- `.github/workflows/ci.yml` — smoke boot now also curls `/api/version` (macOS + Linux).
- `CHANGELOG.md` — new **`## [Unreleased]`** section (this is a backward-compatible feature = a
  future minor; **not tagged/released** — `package.json` stays `1.1.0`. Cut `v1.2.0` when desired).
