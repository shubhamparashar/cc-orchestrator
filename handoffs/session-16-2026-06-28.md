## v1.8.1 — sub-agent `work` field (PR #32)
**The gap (from session-13's follow-up):** a sub-agent's search `body` is a frequency-capped
250-term bag (`SUBAGENT_TERM_CAP`), so a filename mentioned **once** in a tool call could be
evicted — leaving the sub-agent unsearchable by the file it actually worked on. Session docs
never had this problem: they carry an unconditionally-kept `work` field (file basenames, tool
names, leading bash command verbs, spawned sub-agent types, capped at 500 distinct tokens).

**The fix:** give sub-agent docs the same `work` field.
- `lib/subagents.mjs` `docForFile` now sets `work: workString(harvestWorkTokens(text))`.
- `lib/sessionIndex.mjs` `subagentEntries` threads `work: d.work || ''` into the corpus entry.
- `lib/rank.mjs` already scores `work` (`FIELD_BOOST.work = 4`) — **no ranker change needed**.

**The import-cycle break (the reason this wasn't trivial):** `harvestWorkTokens`/`workString`
lived in `sessionIndex.mjs`, which already imports `subagents.mjs`; importing them back the
other way would have created a `sessionIndex ↔ subagents` cycle. So they (plus `WORK_TOKEN_CAP`
and `FILE_PATH_KEYS`) were extracted **verbatim** into a new `lib/work.mjs` that imports only
the leaf `lib/scan.mjs`. Both indexes now import the harvest from the shared leaf — no cycle.

**Tests:** the 5 harvest unit tests moved with the code into new `test/work.test.mjs`;
`test/subagents.test.mjs` gained 2 tests — one proving a tool-only-touched file
(`themeResolver.mjs`) lands in `work` while staying **absent** from `body` (the exact property
fixed), one proving a no-tool sub-agent gets `work === ''`; the exact doc-key-set assertion was
updated to include `work`. Fixture `agent-explore01.jsonl` gained one `tool_use` Read of a file
named only in that call. **132/132 pass**, `node --check` clean, CI green macOS + Linux.
