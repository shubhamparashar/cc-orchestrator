# cc-orchestrator — 2026-06-29 — session 21

**v1.12.0 — persistent live `claude` sessions, started + watched from the dashboard (P1).**
The first phase of a real persistent, remotely-drivable session: the gap left by the headless
one-shot job runner (`claude -p`, buffered, single-turn) and the Mac-only Terminal attach.

## Task 1 — remote-control substrate (findings, no code change)
- **The shim** (`bin/claude-shim` + `bin/install-shim.sh`) is a zsh alias that injects
  `--remote-control` into *interactive* launches. It is **NOT installed** in `~/.zshrc` on this
  machine, and is redundant anyway: `~/.claude/settings.json` already has
  `remoteControlAtStartup: true`, so the real binary starts claude.ai Remote Control natively.
- **`--remote-control` is claude.ai's OWN control channel** (drive a session from claude.ai),
  not cc-orchestrator's. The dashboard's actual control paths today are `sendPrompt` (headless
  `claude -p --resume`, uses `CLAUDE_BIN` + `CLAUDE_NO_RC=1`, bypasses the shim) and
  `attachInTerminal` (Mac/Linux Terminal, local-only).
- **The live `:7433` LaunchAgent** runs `node server.mjs` (not `claude`); it spawns claude on
  demand. No auto-restart on code change — `launchctl kickstart -k` after a merge.
- **Decision: the persistent runner STANDS ALONE.** It spawns the real binary under a PTY and
  drives it directly, reusing only `CLAUDE_BIN` + `CLAUDE_NO_RC=1` from `lib/actions.mjs`. It does
  **not** use `--remote-control`/the shim (that would phone home to claude.ai and isn't our
  control plane).

### PTY without a native dep (the #1 risk)
Interactive `claude` needs a TTY; we allocate one with the OS **`script`** command, not
`node-pty` (zero-dep preserved). Validated empirically before building:
- **Output (P1):** `script -q /dev/null <bin> --resume <id> <flags>` with
  `stdio: ['ignore','pipe','pipe']` works — BSD `script` tolerates a `/dev/null` stdin (a Node
  socketpair stdin fails its `tcgetattr` with ENOTSUP). Linux uses `script -qfc "<cmd>" /dev/null`.
- **Input (P2, validated, not yet wired):** `cat <fifo> | script …` gives `script` an
  *anonymous-pipe* stdin (tolerated) plus a long-lived writable FIFO — dynamic multi-write input
  reaches the pty. (Node socketpair/FIFO-fd stdin both fail; the anonymous pipe is the path.)
- Teardown needs `detached: true` + `process.kill(-pid)` (kill the whole pipeline).

### Security (load-bearing)
- Access level (`ask`/`acceptEdits`/`plan`/`full`) is validated server-side against a **frozen
  allowlist** → a **static flag literal**; a client string is never spliced into the argv.
  `sessionId` stays UUID-validated + `shq()`-escaped (Linux `-c` path). Level lookup is
  proto-pollution-guarded (`__proto__`/`constructor`/`hasOwnProperty` → rejected).
- New POSTs ride the existing `X-CC` CSRF gate + token/auth flow.
- **Full** requires a UI `confirm()` AND a server audit log. The phone IS allowed to start Full
  (server-side, so `IS_REMOTE` does not hide it) — that's the headline use case — which is exactly
  why it must be confirmed, audit-logged, visible as running, and killable.
- Detached process groups are torn down on stop and on shutdown/crash: graceful `stopAllLive`
  (SIGTERM) then `hardKillAllLive` (SIGKILL stragglers whose `exited` flag is unset, so a
  since-reused pid is never signalled), so a restart never orphans an agent.
