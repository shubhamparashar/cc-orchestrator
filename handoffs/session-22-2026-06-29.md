# cc-orchestrator — 2026-06-29 — session 22

**v1.13.0 — start a NEW live session in any repo or worktree (P2a).** Extends the persistent
live-session runner (P1, v1.12.0): "Go live" could only RESUME an existing session; now you can
start a FRESH one in a chosen directory.

## What shipped (P2a)
- **New-session launch.** A **Go live** button in the hero launcher opens `newLiveDialog`: a cwd
  `<select>` of the repos/worktrees the dashboard already knows (worktrees tagged) + an absolute-
  path input override + an access `<select>` + an optional first-prompt textarea (prefilled from
  the launcher input). Starts a fresh persistent `claude` in that directory.
- **`liveId` registry refactor (load-bearing).** A new session has no claude UUID until claude
  creates one, so the registry (`lib/liveSessions.mjs`) is now keyed by a server-generated
  `liveId`; the resumed session id is tracked separately as `sessionId` (`null` for new). Resume
  and new share one registry, viewer, SSE stream, and stop path. `serialize` exposes
  `{ liveId, sessionId, isNew, cwd, level, status, … }`. `/api/live/stop` + SSE `live`/`live-exit`
  key on `liveId`; the frontend (`state.live`, `renderLive`, `openLiveView`, `appendLive`,
  `stopLive`, both SSE listeners) migrated throughout. The dup-guard ("one live per resumed id")
  now only applies to the resume case.
- **`buildClaudeArgs`** builds `[--resume <id>] <flags> [prompt]` for both shapes;
  **`buildScriptSpawn`** takes the generalized `claudeArgs`.

## Security
- **cwd** validated server-side by `isLiveCwd` (absolute, no `..` traversal segment, must be an
  existing directory) and passed only as the `spawn()` cwd option — never a shell token.
- **prompt** (new free-text) — discrete argv item on macOS; `shq()`-escaped inside the `script
  -c` string on Linux (hostile-prompt test included).
- Access level still maps to a STATIC flag literal via the allowlist; `sessionId`/`liveId`
  UUID-validated; new POSTs ride the `X-CC` CSRF + token/auth gate.
- Every start audit-logged by origin (`resume <id>` / `new`), level, cwd, pid — **never the
  prompt**. Full keeps its explicit `confirm()` on both resume and new paths.

## API
- `POST /api/live/start` — resume: `{ sessionId, cwd?, level }`; new: `{ cwd, level, prompt? }`
  (no `sessionId`). Returns the serialized entry incl. `liveId`. cwd validated on both branches.
- `POST /api/live/stop` — `{ liveId }`.
- `GET /api/live` — entries keyed/identified by `liveId`.
